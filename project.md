1.1 아이디어 제안 배경
// 아이디어를 떠올리게 된 계기나 필요성 등
학교 연구실에서 로봇팔을 활용한 프로젝트를 진행하며 VR 기기, Unity, Isaac Sim 등 다양한 툴을 연결하고 원하는 형태로 제어하기까지 수많은 환경 설정 과정을 거쳐야 했다. 컴퓨터공학 전공자인 우리에게 이 과정이 복잡하고 시간이 오래 걸린다고 느꼈고, 이는 자연스럽게 하나의 의문으로 이어졌다. "실제 기업이나 공장에서 로봇을 활용할 때는 어떨까?" 알아본 결과, 중소 제조현장에서는 작업 내용이 바뀔 때마다 전문 SI업체를 별도로 섭외해야 하고, 그 사이 생산이 멈추는 일이 빈번하게 발생하고 있었다. 로봇을 도입했지만 정작 유연하게 활용하지 못하는 것이다. 우리는 이 문제의 핵심이 "로봇을 다루는 언어"가 인간의 언어와 너무 동떨어져 있다는 것이라고 판단했고, LLM을 통해 이 간극을 좁힐 수 있다는 가능성을 발견했다.

1.2 아이디어 핵심 요약
// 제안하는 AI를 통한 솔루션이 무엇인지 핵심 정의
본 팀은 "로봇팔의 ChatGPT" 를 목표로, 자연어 명령만으로 협동로봇을 즉시 제어할 수 있는 SaaS 플랫폼을 제안한다. 핵심 구조는 두 개의 레이어로 이루어진다. 첫째, LLM 통역 레이어다. 사용자가 음성 또는 텍스트로 명령을 입력하면, Gemini Robotics-ER API가 이를 해석해 로봇 제어 코드를 즉시 생성하고 실행한다. 별도의 학습 없이 작동하며, 작업 변경이나 단순 동작 제어는 대부분 이 단계에서 해결된다. 둘째, 정밀 제어 레이어다. LLM 코드 생성만으로 처리하기 어려운 섬세한 물체 조작이나 정밀 위치 파악이 필요한 작업은, OpenVLA 기반 모델을 사용자 환경에 맞게 파인튜닝하여 배포한다. 두 레이어를 연결하는 것은 로봇팔 어댑터 레이어다. 제조사마다 다른 API와 좌표계를 통일된 인터페이스로 추상화하여, 동일한 명령이 Indy7, UR, Franka 등 어떤 로봇팔에서도 동작할 수 있도록 설계한다. 이를 통해 사용자는 로봇의 제조사나 기술 스택에 관계없이 자연어만으로 로봇을 운용할 수 있다.

1.3 아이디어를 통해 해결하고자 하는 문제
// 제안한 AI 솔루션을 통해 개선하고 싶은 불편함이나 문제점을 기술
현재 협동로봇을 도입한 중소 제조현장이 겪는 핵심 문제는 "로봇을 샀지만 자유롭게 못 쓴다" 는 것이다. 작업 내용이 바뀔 때마다 전문 SI업체를 섭외해야 하고, 그 사이 생산라인은 멈춘다. 로봇 전문 인력을 내부에 두기엔 인건비 부담이 크고, 그렇다고 매번 외주를 맡기자니 비용과 대기 시간이 로봇 도입 효과를 반감시킨다. 결국 수천만 원짜리 로봇이 단 하나의 반복 작업만 수행하는 고정 기계로 전락하는 경우가 많다. 이 문제의 근본 원인은 로봇을 다루는 언어와 사람의 언어 사이의 간극이다. 로봇은 좌표, 관절 각도, 제조사별 API로 이야기하지만, 현장 작업자는 "저 부품 집어서 오른쪽 박스에 넣어"라고 말한다. 이 간극을 메우기 위해 지금까지는 사람(SI업체, 로봇 엔지니어)이 필요했다. 본 팀은 이 역할을 AI가 대신할 수 있다고 판단한다. 자연어 명령을 로봇 제어 코드로 즉시 변환하는 LLM 기반 인터페이스를 제공함으로써, 전문 인력 없이도 현장 작업자가 직접 로봇의 작업을 바꾸고 운용할 수 있는 환경을 만드는 것이 본 아이디어가 해결하고자 하는 문제다.

자연어로 로봇을 제어하려는 시도는 이미 글로벌 시장에서 시작되고 있다. Mbodi(YC 스타트업)는 자연어 프롬프트로 AI 에이전트 클러스터가 로봇을 학습시키는 방식으로, 매일 바뀌는 포장 작업처럼 재프로그래밍이 불가능한 환경을 타겟으로 개발 중이다. Physical Intelligence(π0)는 7종 로봇, 68개 태스크로 사전학습된 범용 로봇 파운데이션 모델을 개발하고 있으며, Zeon은 과학자가 영어로 실험을 입력하면 로봇팔이 실행하는 연구실 특화 서비스다. Daaramall
그러나 이 서비스들은 공통적으로 특정 로봇 또는 특정 환경에 종속되어 있다는 한계가 있다. Mbodi는 자체 하드웨어 생태계 중심이고, π0는 연구 수준의 범용 모델로 실제 현장 배포까지 갈 길이 멀며, Zeon은 연구실 환경에 국한된다. 국내에는 이에 준하는 서비스 자체가 존재하지 않는다.
본 팀의 차별점은 다음 세 가지다.
첫째, 제조사 독립적인 어댑터 레이어다. Indy7, UR, Franka 등 서로 다른 API와 좌표계를 가진 로봇팔을 하나의 인터페이스로 통합한다. 사용자는 어떤 로봇팔을 쓰든 동일한 방식으로 명령을 내릴 수 있다.
둘째, 즉시 사용 가능한 기본 티어와 커스텀 학습 프리미엄 티어의 이중 구조다. Gemini Robotics-ER 기반의 기본 티어는 별도 학습 없이 즉시 작동하고, LLM으로 해결되지 않는 정밀 조작은 OpenVLA 파인튜닝을 통한 프리미엄 티어로 커버한다. 사용자는 필요에 따라 두 레이어를 선택적으로 활용할 수 있다.
셋째, 국내 협동로봇 시장 최초의 자연어 제어 SaaS라는 포지셔닝이다. 글로벌 서비스들이 영어 기반, 해외 로봇 중심으로 개발되는 사이, 국내 1위 협동로봇인 Indy7을 첫 타깃으로 한국어 명령 처리와 국내 제조 현장에 특화된 서비스를 선점한다.

3.2 도전과제 및 해결방안
// 프로젝트를 진행하면서 마주칠 것으로 예상되는 난관은 무엇이며, 이를 어떻게 극복할 계획(ex. 스터디, 멘토링 등)인지

난관 1: AI가 만든 계획이 현실에서 작동하지 않는 문제
AI는 "물건을 집어서 옮겨라"라는 훌륭한 계획을 세울 수 있지만, 실제 로봇팔이 그 위치에 물리적으로 닿지 못하거나, 카메라가 인식한 위치와 실제 위치가 수 센티미터 어긋나는 문제가 발생합니다. 사람이 눈으로 보고 손을 뻗으면 자연스럽게 보정하는 과정이, 로봇에게는 아직 어렵습니다.
극복 계획: 코드를 실행하기 전에 "이 동작이 물리적으로 가능한가?"를 자동 검증하는 안전 장치를 넣고, 실행 후에도 카메라로 결과를 확인하여 실패 시 자동 재시도합니다. 대회에서 제공되는 전문가 멘토링을 통해 이 검증 시스템의 정확도를 높이겠습니다.

난관 2: 여러 기술을 동시에 처음 다루는 도전
이 프로젝트는 AI, 로봇 하드웨어, 클라우드 통신, 카메라 처리를 한꺼번에 개발해야 합니다. 하나만 해도 어려운 영역들을 동시에 처음 다루기 때문에, 어디서 문제가 생겼는지 파악하는 것 자체가 난관입니다.
극복 계획: 처음부터 실물 로봇으로 개발하지 않고, 가상 시뮬레이션(Isaac Sim)에서 먼저 모든 기능을 검증합니다. 시뮬레이션에서 되는 것이 실물에서 안 되면 하드웨어 문제, 시뮬레이션에서도 안 되면 소프트웨어 문제로 원인을 빠르게 좁힐 수 있습니다. 또한 대회에서 제공되는 워크샵과 세미나를 통해 전문가 및 타 팀과 기술 교류를 진행하여 병목을 조기에 해결하겠습니다.

난관 1: AI가 생성한 코드가 실제 물리 환경에서 작동하지 않는 문제
LLM은 "물건을 집어서 옮겨라"는 명령에 대해 논리적으로 완벽한 코드를 생성할 수 있지만, 실제 실행 환경에서는 카메라 캘리브레이션 오차로 인한 위치 불일치, 로봇팔의 작업 반경 초과, 그리퍼 파지 각도 오류 등 다양한 물리적 제약에 부딪힌다. 특히 카메라 좌표계와 로봇 좌표계 사이의 변환(eye-to-hand calibration)은 수 밀리미터의 오차도 파지 실패로 이어질 수 있다.
극복 계획: 코드 실행 전 역기구학(IK) 해 존재 여부와 충돌 가능성을 Isaac Sim에서 자동 검증하는 사전 검증 레이어를 구현한다. 실행 후에는 RealSense 카메라로 결과를 확인하여 실패 시 파라미터를 조정하며 자동 재시도하는 피드백 루프를 구성한다. 캘리브레이션 정확도는 ChArUco 보드 기반 캘리브레이션을 주 1회 수행하여 유지한다. 대회 멘토링을 통해 현장 전문가의 캘리브레이션 노하우를 빠르게 흡수할 계획이다.
난관 2: 자연어 명령의 모호성으로 인한 오작동 위험
"오른쪽 부품 집어"라는 명령은 사람에게는 명확하지만, LLM 입장에서는 어떤 부품인지, 얼마나 오른쪽인지, 어디에 놓을지가 불명확하다. 모호한 명령을 그대로 실행하면 엉뚱한 동작이 나오고, 제조 현장에서는 이것이 안전사고로 이어질 수 있다.
극복 계획: LLM이 명령을 파싱한 뒤 실행 전에 "오른쪽 컨베이어 위 빨간 볼트를 집어서 왼쪽 박스에 넣는 것이 맞습니까?"처럼 의도를 재확인하는 confirmation step을 기본으로 넣는다. 명령의 완성도를 점수화하는 confidence threshold를 설정하여, 일정 수준 이하의 모호한 명령은 자동으로 추가 질문을 요청하도록 설계한다. 초기 4주간 다양한 모호 명령 케이스를 직접 수집하여 프롬프트 엔지니어링으로 정확도를 높이는 스터디를 진행한다.

난관 3: 제조사별 로봇 API 파편화 문제 (어댑터 레이어 개발)
Indy7(IndySDK), UR(URScript), Franka(libfranka)는 각각 좌표계, 관절 제어 방식, 통신 프로토콜이 완전히 다르다. LLM이 생성한 추상적인 명령을 각 제조사 API에 맞게 변환하는 어댑터 레이어를 직접 개발해야 하며, 이 레이어가 불안정하면 서비스 전체가 흔들린다.
극복 계획: 대회 기간 내에는 Indy7 단일 어댑터에 집중하여 안정성을 확보한 뒤, UR 어댑터를 추가하는 순서로 진행한다. ROS2의 표준 인터페이스(MoveIt2)를 중간 계층으로 활용하면 제조사별 차이를 ROS2 레벨에서 흡수할 수 있어 어댑터 개발 부담을 줄일 수 있다. 뉴로메카 공식 문서와 GitHub 커뮤니티를 적극 활용하고, 막히는 부분은 대회 멘토링을 통해 해결한다.

4. 기대효과 및 성장목표
   ※ 작성 포인트: 이 프로젝트가 세상에 미칠 긍정적인 변화와 팀원들의 성장을 기술

4.1 기대효과
// 이 아이디어가 실제로 구현되었을 때 사회적 산업적 기대효과가 있을지
현재 협동로봇 시장은 빠르게 성장하고 있지만, 정작 현장에서 로봇을 자유롭게 활용하지 못하는 중소 제조업체가 대다수다. 본 서비스가 실제로 구현되었을 때 기대할 수 있는 효과는 다음과 같다.
산업적 측면에서, 작업 변경 시마다 발생하던 SI업체 의존 비용과 대기 시간이 대폭 감소한다. 현장 작업자가 자연어 명령만으로 로봇의 작업을 즉시 전환할 수 있게 되어, 다품종 소량생산 환경에서의 로봇 활용도가 크게 높아진다. 나아가 Indy7·UR·Franka 등 다양한 로봇팔을 하나의 인터페이스로 통합 운용할 수 있는 플랫폼으로 발전할 경우, 제조업 전반의 자동화 접근성이 낮아지는 효과를 기대할 수 있다.
사회적 측면에서, 로봇 전문 인력이 없는 중소기업도 AI를 매개로 로봇 자동화에 진입할 수 있는 환경이 마련된다. 이는 대기업과 중소기업 사이의 자동화 격차를 줄이는 데 기여한다. 또한 본 기술은 제조 현장에 국한되지 않고, 거동이 불편한 사용자가 로봇팔을 음성으로 제어하는 보조 도구로 확장될 가능성을 가지며, 장애인의 일상 자립을 지원하는 방향으로도 발전할 수 있다.

4.2 성장목표
// 이번 대회를 통해 팀원들이 미래 AI 인재로서 어떤 역량을 키우고 싶은지
기본적으로 모든 팀원은 서로의 내용을 함께 공부하고 공유하지만 구체적으로, 팀원 A (로봇 제어) 는 Isaac Sim에서 시뮬레이션으로만 다뤄보던 로봇 제어를 실제 Indy7 하드웨어와 연결하며 처음으로 "내가 짠 코드가 실제 물체를 움직이는" 경험을 한다. ROS2 기반 실시간 제어와 캘리브레이션 오차 처리 등 교과서에 없는 현장형 문제를 직접 해결하며 로봇 소프트웨어 엔지니어로서의 실전 감각을 키운다. 특히 AI가 생성한 제어 코드가 실제 환경에서 예상대로 작동하지 않을 때, 단순히 AI에게 재질문하는 것에 그치지 않고 왜 틀렸는지를 물리적으로 분석하고 파라미터를 직접 조정하여 원하는 동작을 만들어내는 능력을 기른다.
팀원 B (AI/백엔드) 는 LLM을 단순히 챗봇으로 쓰는 것을 넘어, 실제 물리 시스템을 제어하는 두뇌로 연결하는 경험을 한다. Gemini API와 OpenVLA를 직접 연동하고, 모델이 틀린 코드를 생성했을 때 프롬프트를 어떻게 수정해야 하는지, 파인튜닝 데이터를 어떻게 구성해야 정밀도가 올라가는지를 수십 번의 실험을 통해 직접 체득한다. AI를 "쓰는" 수준이 아니라 "다루고 조율하는" 수준으로 끌어올리는 것이 목표다.
팀원 C (프론트/인프라) 는 로봇을 한 번도 다뤄보지 않은 현장 작업자가 우리 웹 인터페이스만으로 로봇을 움직이는 장면을 직접 만들어낸다. 기술이 아무리 좋아도 UI가 복잡하면 아무도 안 쓴다는 것을 몸으로 배우며, 사용자 테스트를 반복하면서 인터페이스를 조금씩 개선해나가는 경험을 쌓는다. 동시에 클라우드 서버와 엣지 디바이스 사이의 통신 구조를 직접 설계하고 운용하며 AI 서비스 인프라 전반을 다루는 역량을 키운다.
세 명 모두가 공통적으로 얻고자 하는 것은 하나다. AI가 일상이 된 시대에, AI한테 질문하고 답을 받아 쓰는 것을 넘어 AI를 직접 연결하고, 한계를 파악하고, 원하는 방향으로 조율하는 경험이다. 앞으로 개발 현장에서 살아남는 사람은 AI를 도구로 능숙하게 다룰 줄 아는 사람이라는 것을 이 프로젝트를 통해 직접 증명하고자 한다.
